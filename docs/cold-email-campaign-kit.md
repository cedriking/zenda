## Email Verification Pipeline

Every prospect email **must** pass verification before sending. No exceptions.

### Step 1: Gather emails from website contact pages only

- Only use emails found on the business's own website (Contacto, Nosotros, footer)
- Never guess or infer emails (e.g., info@, contacto@ without seeing it on their site)
- Google Maps listings alone are not sufficient — must verify on actual website

### Step 2: Run the verifier

```bash
# Single email
echo "contacto@example.com" | bun run scripts/verify-prospect-emails.ts

# Batch from file
bun run scripts/verify-prospect-emails.ts prospects.txt
```

### Step 3: Only send to `valid` results

| Status | Action |
|--------|--------|
| `valid` | Safe to send |
| `risky` | Do not send — likely catch-all or low quality |
| `invalid` | Do not send — will bounce |
| `unknown` | Do not send — cannot confirm deliverability |

### Step 4: Log verification status in prospect list

Add a `Verified` column to the prospect tracking table with date and method.

### Why this exists

The first prospect list had 75% bounce rate on inferred emails (commit 9f26ab1). Only website-verified emails are acceptable. The 2-tier verifier catches:
- Syntax errors
- Disposable/temp email domains
- Domains with no MX records (will bounce)
- Optionally: SMTP-level validation via AbstractAPI (if `ABSTRACT_API_KEY` env var is set)
